## My Reflection: CV

Before this task I had limited experience with Markdown. I knew it existed but had never used it to structure a full document like a CV. This assignment gave me hands-on exposure to how Markdown works as a lightweight markup language and how it can be used to present information in a clean, readable format. Getting started was straightforward. Markdown uses simple symbols like # for headings, | for tables, --- for dividers, and ** for bold text, which made it easy to pick up quickly. I used these elements to organise my CV into clear sections; Objective, Education, Experience, Technical Skills, Academic Results, and References. The table syntax was particularly useful for displaying my matric results and CPUT academic results in a structured way. One challenge I encountered was the limitation around styling. Unlike Word or HTML, Markdown does not support colors or custom fonts natively. My original CV had a distinctive dark navy and yellow color scheme, but in Markdown those visual elements could not be replicated. This made me realise that Markdown is best suited for content structure and readability rather than visual design. Overall, I found Markdown to be a practical and efficient tool, especially for developer environments like GitHub, where it renders cleanly. It taught me to think about content separately from presentation, which is actually a valuable principle in software development similar to separating logic from the UI. I now feel confident using Markdown for documentation, README files, and portfolio content going forward.

## My Reflection: Mock Interview Video

As part of my GitHub portfolio assignment, I was required to embed a mock interview video into my Markdown CV hosted on GitHub Pages. This turned out to be one of the more challenging parts of the task, and I learned a lot from the process.
My original video was in .mov format, which is recorded on Apple devices. The first issue I encountered was that .mov files are not browser-compatible, meaning the video would not play directly in a web browser. To solve this, I had to convert the file from .mov to .mp4, which is a widely supported video format across all browsers. Another limitation I discovered is that pure Markdown does not support direct video embedding. There is no native Markdown syntax like there is for images. To work around this, I used an HTML <video> tag inside the Markdown file which GitHub Pages supports since it renders Markdown as HTML. This experience taught me that Markdown has limitations when it comes to rich media, and that sometimes you need to mix in HTML to achieve what you need. It also gave me practical experience with file formats and browser compatibility, concepts that are directly relevant to web development. Overall embedding the video pushed me to problem-solve beyond basic Markdown and think like a developer.

## My Reflection: GitHub Pages

As part of my portfolio assignment, I was required to deploy my Markdown CV using GitHub Pages. This was my first time setting up a live website directly from a GitHub repository, and it turned out to be a valuable and eye-opening experience.
The first step was creating a repository named hillaryitlhabanyeng.github.io which had to match my GitHub username exactly. I learned that this specific naming convention is what tells GitHub to treat the repository as a personal GitHub Pages site. Any deviation from this format would mean the site would not deploy correctly. Once the repository was set up, I pushed my Markdown CV file as the README.md so that GitHub Pages could pick it up and render it as a webpage. Seeing a plain Markdown file transform into a live, accessible website at a public URL was a satisfying moment that made the concept of static site hosting very real for me.

One thing that stood out was how GitHub Pages bridges the gap between version control and web hosting. Every time I pushed a change to the repository, the live site updated automatically. This gave me a practical understanding of continuous deployment on a small scale, which is a concept widely used in professional software development. Overall GitHub Pages taught me that deploying a website does not always require a backend server or complex infrastructure. It showed me how developers can use tools they already work with like Git and GitHub to share their work with the world quickly and efficiently. This is a skill I will definitely carry forward into my career.
